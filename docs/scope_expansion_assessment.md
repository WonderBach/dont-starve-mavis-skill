# 从首冬扩到一年四季的评估

## 结论

建议扩。

但扩法不是“做全游戏数据库”，而是把 MVP 从“首冬规划 Agent”升级成“新手四季生存规划 Agent”。首冬仍然作为冬季重点子模块保留。

## 为什么扩是合理的

之前只做首冬，有一个很强的产品切口：新手最容易在入冬后被低温、断粮、长夜、鹿角怪和猎犬连续击穿。

现在如果入口变成单机版和联机版都覆盖，而且用户可能在任意时间打开 Agent，只做首冬会有两个问题：

- 用户春天被雨、潮湿、青蛙雨和兔子洞关闭卡住时，Agent 接不住。
- 用户夏天被过热、自燃、食物腐烂和基地烧毁卡住时，Agent 会显得范围过窄。
- 用户刚开局或刚换季时，真正需要的是“当前季节最低保命路线”，不一定是首冬路线。

所以一年四季更适合现在的入口形态。

## 工作量判断

难度：中等偏低。

原因是现有首冬库已经有骨架：

- 事实库
- 规则层
- 用户分流卡片
- 停尸房 OCR 死因推理
- 固定输出结构
- 控制台援助代码包

扩到一年四季主要增加三块：

- 春季：雨、潮湿、雷击、青蛙雨、兔子洞关闭、发情牛群、蜜蜂攻击性上升。
- 夏季：过热、冷火/冷暖石、防暑装备、野火/自燃、食物腐烂加速、植物枯萎。
- 秋季：从“默认开局”升级为“建设窗口”，重点是为下个季节建立最低闭环。

这不是指数级扩张。只要不碰全角色、全食谱、洞穴远古、全 Boss、联机专属团队分工，范围仍然可控。

## 推荐的新框架

四季生存能力不再只写成首冬三链，而是：

> 四季生存能力 = 通用生存链 + 当季环境链 + 过季准备链

通用生存链：

- 食物链
- 光源链
- 基础战斗安全链
- 基地安全链

当季环境链：

- 秋季：建设链，核心是科技、基地、锅、火源、武器护甲。
- 冬季：保暖链，核心是暖石、火源、御寒装备、燃料。
- 春季：雨季/潮湿链，核心是雨具、避雷、替代食物路线、理智压力控制。
- 夏季：降温链 + 防火链，核心是冷火、冷暖石、防暑装备、灭火/避开自燃。

## 产品边界

建议产品文档改成：

> 首版以《饥荒》单机 RoG/Giant Edition 默认世界的一年四季基础生存为主，兼容裸 DS 和 DST 的基础差异；不覆盖年节活动、全角色打法、完整食谱、高级 Boss、洞穴远古和联机专属复杂机制。

这样比“只做首冬”更完整，也比“做全饥荒百科”更安全。

## 对现有文件的影响

- `data/rules.json` 保留：作为冬季/首冬重点规则。
- `data/seasonal_survival.json` 新增：作为四季总规则入口。
- `data/morgue_ocr_rules.json` 后续可扩：新增 Overheating、Fire/Burning、Frog、Lightning 等季节死因映射。
- `data/agent_cards.json` 后续可扩：把“快到冬天”改成“快换季/已经入季/反复死在某季”。

当前已经新增四季总规则，后续若进入产品实现，优先让 Agent 调用 `seasonal_survival.json` 做入口，再按季节下钻到首冬规则。
