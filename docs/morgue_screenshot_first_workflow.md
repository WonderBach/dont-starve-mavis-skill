# 取材料引导优先与可选自动获取工作流评估

## 结论

这个 Skill 的默认入口应从“系统先尝试获取死亡记录”调整为“先取材料”。用户只说“帮我看看饥荒怎么活得更久”时，当前缺少死亡记录或局内信息，第一轮应该引导用户选择材料来源，而不是直接诊断。

推荐主流程：

1. 用户触发 Skill。
2. Mavis 判断是否已有截图或文字死亡材料。
3. 没有材料时，先给取材料引导：上传截图、文字描述、或可选自动获取死亡记录。
4. 上传截图后，Mavis 读取截图，OCR 出 `Days Lived` 和 `Cause of Death`。
5. 文字描述后，Skill 用最多两轮问题补齐第几天、怎么死、缺什么资源。
6. 用户明确选择自动获取后，APP Agent 才检测环境并尝试获取死亡记录截图。
7. 自动获取成功后回到截图分析；失败后回到取材料引导。
8. 饥荒 Skill 根据天数和死因做推理，输出下一局最低修正方案。

## 工作量评估

难度：中等偏低。

如果 Mavis 已经有图片理解能力，Skill 侧不需要自己做 OCR 模型，只需要写清楚输入输出契约和推理规则。主要工作量在四处：

- 取材料引导：首轮给出清晰的三选一入口。
- 视觉抽取：Mavis 能否稳定读出表格里的 `Days Lived` 和 `Cause of Death`。
- 文字诊断：用两轮以内问题从用户描述中补齐关键链状态。
- 可选自动获取：用户明确选择后，APP Agent 能否稳定获取死亡记录截图。

取材料优先的优势是：

- 不会在用户没有提供材料时贸然进入本地环境操作。
- 用户能用截图、文字或可选自动获取三种方式进入复盘。
- 自动获取失败后仍能自然回到截图上传或文字描述。

风险是：

- 用户文字描述可能不完整，需要控制补问轮数。
- 游戏界面语言、分辨率、字体、遮挡会影响 OCR。
- 停尸房可能有多条记录，需要决定最近一次和历史模式的关系。
- `Cause of Death` 是最后一击，不等于根因，不能只机械翻译。

## 三层分工

### APP Agent

负责可选自动获取动作，不负责推理。

它只在用户明确选择自动获取或明确要求获取死亡记录时做：

- 检测 Steam 和《饥荒》单机版 / 联机版安装状态。
- 在环境允许时尝试进入 Morgue / 停尸房 / 死亡记录界面。
- 截图并把图片交给 Mavis。
- 任一环节失败时停止继续操作，回到取材料引导。

它不需要理解“为什么死”，也不需要判断下一局怎么改。

### Mavis 本体

负责对话编排、读图和 OCR。

它应该输出结构化字段：

```json
{
  "days_lived": 23,
  "deceased": "Wilson",
  "cause_of_death": "Darkness",
  "mode": "Survival",
  "ocr_confidence": 0.92,
  "raw_ocr_text": "..."
}
```

最低成功条件是读出：

- `days_lived`
- `cause_of_death`

读不出 `deceased` 或 `mode` 可以降置信，但不一定阻塞。

### 饥荒 Skill

负责推理和输出。

它应该做：

- 先复述材料结果：第几天、死因或用户描述的死亡情况。
- 用天数判断阶段或季节，但只作为辅助判断。
- 用死因映射根因链。
- 给下一局最小修正方案。
- 根据用户介入偏好，决定是否给控制台代码。

## 最小输出模板

```text
我读到这条死亡记录：
- 生存天数：第 23 天
- 停尸房死因：Darkness
- 阶段判断：大概率处于首冬窗口

这不是单纯“血量不够”，更像是光源链断了：
冬季夜晚更长，移动光源或燃料不足，最后被黑暗/Charlie 结算。

下一局最低修正：
1. 第 10 天前做火坑和锅。
2. 第 15-20 天整理木头、草、树枝。
3. 出门至少带 2 根火把或等价材料。
4. 冬季夜晚不远行。

我暂时不直接展示其他更强、可以直接修改游戏数值的代码。
如果你觉得当前难度还是太高，确实需要的话，也可以再问我要。
```

## 规则落点

当前数据库已对应调整：

- `SKILL.md`：取材料引导、截图分析、文字问答诊断、可选自动获取路径。
- `data/morgue_ocr_rules.json`：截图 OCR、取材料策略、可选自动获取回退。
- `data/agent_cards.json`：三种材料来源入口和动态补问。
- `data/seasonal_survival.json`：四季生存链和季节死因映射。
