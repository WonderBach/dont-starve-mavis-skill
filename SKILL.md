# Mavis Skill：饥荒死亡复盘 / 生存复盘

## 目标

当用户想知道“我在《饥荒》里怎么活得更久”或“为什么总是死”时，默认先进入取材料引导：第一轮不分析、不分析角色，直接给出三张可点击卡片，让用户选择提供哪类死亡材料或局内信息，再进入截图分析、选择题诊断，或在用户明确选择时尝试自动获取死亡记录。

Skill 的核心输出是：

- 生存天数
- 停尸房直接死因
- 大致阶段/季节判断
- 根因链判断
- 下一局最低修正方案
- 可选援助：攻略、物资代码、属性/开挂代码

截图上传、选择题诊断和可选自动获取是三种并列材料来源，开场以三张卡片呈现，用户点击即可，不必打字。自动获取死亡记录不是默认入口，只在用户明确选择或明确要求打开游戏/进入停尸房/获取死亡记录时启用。死亡分析、OCR、根因链和攻略输出能力保持原有结构；全程不分析角色/头像。

## 触发语义

当用户表达以下任一意图时，启用本 Skill：

- “帮我看看饥荒怎么活得更久”
- “帮我看看饥荒怎么死的”
- “分析停尸房/讣告/死亡记录”
- “打开饥荒看看我死因”
- “我又死在冬天/春天/夏天了”
- 用户上传《饥荒》停尸房截图
- 用户询问《饥荒》死亡原因、生存天数、下一局怎么改

如果用户只是泛泛问攻略，不需要分析个人死亡记录，可以走普通攻略路线；如果用户触发的是本 Skill/case，但当前没有提供死亡记录、截图或局内信息，先进入取材料引导，不要直接分析，也不要默认启动 App Agent。

只有以下情况才启动 App Agent：

1. 用户明确选择“尝试自动获取死亡记录”。
2. 用户明确表达“帮我打开游戏 / 帮我进停尸房 / 帮我获取死亡记录 / 自动帮我找记录”。

除此之外，不要检测 Steam，不要打开游戏，不要进入停尸房或讣告页面。

## 总体原则

1. 取材料引导优先：用户只说想活得更久但没有材料时，先给三张卡片引导其选择截图、选择题诊断或可选自动获取。
2. 截图路径：Mavis 负责读图 OCR，本 Skill 负责推理。
3. 选择题路径：季节 / 数值 / 卡点三张卡片点选，汇总成画像后做低/中置信诊断，不做自由文字长问卷；仅在冲突或都说不清时补问一句。
4. 可选自动获取路径：App Agent 负责环境检测和尝试获取死亡记录截图；成功后回到截图分析，失败后回到取材料引导。
5. 先输出可从材料读到的硬信息：生存天数、直接死因或用户描述的死亡情况。
6. 再解释“直接死因”和“根因链”的区别。
7. 用户介入能力保留，但放在诊断之后。

## 角色分工

### Mavis 本体

负责对话编排、图片理解和 OCR。

Mavis 应把截图 OCR 成结构化字段：

```json
{
  "days_lived": 23,
  "deceased": "Wilson",
  "cause_of_death": "Darkness",
  "mode": "Survival",
  "ocr_confidence": 0.92,
  "raw_ocr_text": "..."
}
```

最低成功条件：

- `days_lived` 可读
- `cause_of_death` 可读

`deceased` 和 `mode` 可选，读不到时降置信但不阻塞。

### App Agent

仅在用户明确选择自动获取时，负责本地应用检测、打开游戏、导航到停尸房/讣告页面并截图。

App Agent 不负责解释死因。

### 本 Skill

负责：

- 规定调用顺序
- 校验 OCR 字段
- 用天数判断阶段
- 用死因映射根因链
- 输出攻略或控制台援助

## 主流程

### 0. 用户触发

用户触发 Skill 后，先判断当前是否已经有死亡材料或局内信息：

- 如果用户已经上传停尸房/讣告/死亡结算截图：进入截图分析流程。
- 如果用户已经描述第几天死、怎么死、缺什么资源或当前季节症状：进入选择题诊断（可直接用文字里的信息替用户预填卡片）。
- 如果用户明确要求自动获取死亡记录：进入可选自动获取流程。
- 如果用户只是说“帮我看看饥荒怎么活得更久”或类似泛化需求，且没有提供死亡记录或局内信息：进入取材料引导，给出三张卡片。

推荐开场话术（以三张卡片呈现，用户点击即可，不必打字）：

>

> 把停尸房 / 讣告 / 死亡结算截图丢给我，我直接读出你活了几天、死因是什么，复盘最准。
>
> **📝 做几道选择题，我帮你复盘**
> 不想截图也行，点几下告诉我卡在哪个季节、哪个数值保不住、最难的是哪一步，我照样帮你诊断。
>
> **🎮 让我自动去翻你的死亡记录**
> 我可以试着自己打开游戏、翻出停尸房截图——但**只支持 Steam 版**，还得请老板配合、耐心等自动化跑完，也**有一定概率翻车**；万一失败，随时能退回截图或选择题。
>
> <small>（小声：我这套复盘主要覆盖《饥荒》经典版的四季生存，DLC 和联机版特殊活动…小马暂时还没吃透哈~）</small>

在三张卡片下方固定附一行小字版本声明（上面这句），说明覆盖范围是《饥荒》经典版四季，不含 DLC 与联机版特殊活动。

这个 case 的第一轮回复不要直接分析，不要分析角色，也不要启动 App Agent。

### 1. 取材料引导

取材料引导是默认入口，适用于用户只提出“想活得更久”“总是死”“帮我复盘”但没有给材料的场景。以三张卡片呈现，用户点击即可。

分流：

- 上传截图：进入截图分析规则。
- 做几道选择题：进入选择题诊断路线（季节 / 数值 / 卡点三张卡片点选，选完自动汇总分析，不做自由文字长问卷）。
- 尝试自动获取：进入 App Agent 可选自动获取路径。

### 2. 可选自动获取：检测 Steam

仅当用户明确选择自动获取或明确要求打开游戏/进入停尸房/获取死亡记录时，App Agent 才检测 Steam 是否安装。

优先检测：

- macOS：`/Applications/Steam.app`、`~/Applications/Steam.app`
- Windows：常见 Steam 安装路径和注册表
- Steam library manifest：`steamapps/libraryfolders.vdf`
- Steam URL scheme：`steam://`

结果分流：

- 未检测到 Steam：不要继续找游戏包名，进入非 Steam 兜底。
- 检测到 Steam：继续检测游戏安装。

未检测到 Steam 时的推荐话术：

> 我没有检测到 Steam 版《饥荒》的可自动获取入口。你可以上传停尸房/讣告截图，我照样能分析；如果不想截图，也可以做几道选择题（选一下卡在哪个季节、哪个数值保不住、最难的是哪一步），我帮你复盘。

### 3. 可选自动获取：检测 Steam 游戏安装

Steam 版游戏以 app id 为准，不靠模糊包名。

目标游戏：

| 游戏 | Steam app id | 用途 |
| --- | --- | --- |
| Don't Starve | `219740` | 单机版 |
| Don't Starve Together | `322330` | 联机版 |

检测方式：

1. 从 Steam 的 `libraryfolders.vdf` 找所有库目录。
2. 查找 `appmanifest_219740.acf` 和 `appmanifest_322330.acf`。
3. 如果 manifest 存在，认为该 Steam 游戏已安装或至少在库中有可用入口。
4. 如果 manifest 不存在，不要猜非白名单包名。

结果分流：

- 只检测到 `219740`：走单机版可选自动获取路径。
- 只检测到 `322330`：走联机版可选自动获取路径。
- 两个都检测到：问用户要分析哪一个。
- 两个都没检测到：回到取材料引导，请用户上传截图或做选择题。

两个都安装时的推荐话术：

> 我检测到你同时装了《饥荒》单机版和《饥荒联机版》。你想分析哪一个的死亡记录？

### 4C. 自动化截图与识别方式

进入停尸间 / 讣告（死因界面）后，统一使用 `python_executor` + `pyautogui` 截图，再交给 `analyze_image` 分析。这种方式不依赖屏幕缩放比例，在非 100% DPI 环境下也能正常截全图。

截图脚本：

```python
import pyautogui, pygetwindow as gw, time
win = gw.getWindowsWithTitle("目标窗口标题")[0]
if win.isMinimized:
    win.restore()
win.activate()
time.sleep(0.5)
pyautogui.screenshot().save("输出路径.png")
```

- 「目标窗口标题」：单机版填「Don't Starve」，联机版填「Don't Starve Together」。
- 「输出路径.png」替换为实际截图保存路径。

识别只做两件事（针对死因界面）：

1. **存活天数**（Days Lived）
2. **死因**（Cause of Death）

**不分析角色（Deceased）**：角色/头像在截图里识别不稳定，不作为分析字段。

分析完成后，由 Mavis 呈现「死因 + 存活天数」，并给出对应攻略。

## 截图分析规则

Mavis OCR 后，本 Skill 按以下顺序分析：

1. 判断截图是否为停尸房/讣告页面。
2. 抽取所有可见死亡记录，不只读最新一条：`Days Lived`、`Cause of Death`、`Mode`（不分析角色/头像，图片识别不稳定）。
3. 先做综合画像：统计死亡天数集中在哪些区间、重复出现的死因、重复断裂的根因链。
4. 判断“最近一次死亡”和“历史高频问题”是否一致：最近一次用于解释当下，历史模式用于给下一局优先级。
5. 输出综合结论：用户大概常死在什么时候、最常见死因是什么、背后的主要断链是什么。
6. 再拿综合结论查 `death_causes.json`、`seasonal_survival.json`，必要时下钻 `rules.json`。
7. 生成下一局最低修正方案：优先解决高频/早期/最阻断进度的死因，而不是只修最新一条。
8. 根据用户介入策略输出正常攻略路线和对应物资补救代码。

多记录分析原则：

- 如果截图有多条记录，必须先列出可读记录摘要，再做综合判断。
- 不要默认只分析最上方/最新记录；最上方记录只是“最近一次死亡”。
- 如果某类死因重复出现，优先级高于单次偶发死因。
- 如果死亡天数集中在同一窗口，优先判断该阶段的系统性短板。
- 如果最新一次和历史高频不一致，输出时分开说：“最近一次是 X，但历史上更常见的问题是 Y。”
- 如果 OCR 只读清部分记录，也要说明“基于可读记录分析”。

规则表调用顺序必须固定：

1. 先形成多记录综合结论：死亡天数分布、死因分布、根因链分布。
2. 再查 `data/death_causes.json`：用综合后的高频/最近 `Cause of Death` 确定直接死因、根因链、季节风险和基础补救方向。
3. 再查 `data/seasonal_survival.json`：根据根因链和天数/季节上下文，补充四季层面的环境链判断。
4. 只有当死因或上下文强相关冬季/首冬时，才下钻 `data/rules.json`：例如 `Freezing`、`Deerclops`、冬季长夜黑暗、冬季断粮、蓝猎犬、暖石/冬帽/火源/燃料问题。
5. 不要把 `data/rules.json` 当成总入口；它只是冬季放大镜。

截图不是停尸房时：

> 这张图不像停尸房/讣告页面，我读不到稳定的生存天数和死因。你可以进入停尸房再截一张；或者我用手动问题帮你判断。

OCR 低置信时：

- 最多补问 1 个问题。
- 不进入长问卷。

推荐补问：

> 我能看出生存天数大概是第 X 天，但死因那一栏不太清楚。你能确认一下 Cause of Death 写的是什么吗？

## 天数到阶段的粗略判断

仅当默认采用 RoG/Giant Edition 秋季起始时使用：

| 生存天数 | 推断阶段 |
| --- | --- |
| 1-5 | 开局基础生存失败 |
| 6-14 | 秋季建设窗口 |
| 15-20 | 首冬准备窗口 |
| 21-36 | 冬季/首冬窗口 |
| 37-56 | 春季雨湿窗口 |
| 57-72 | 夏季过热/防火窗口 |
| 73+ | 后续循环或后期生存 |

注意：

- RoG 可能春季起始。
- DST 季节长度和单机有差异。
- 用户可能改世界设置。
- 裸 Don't Starve 本体不是四季制。

因此：天数只做辅助，最终以死因和截图上下文为主。

## 死因到根因链

| Cause of Death | 直接含义 | 根因链 |
| --- | --- | --- |
| Freezing / Frozen | 冻死 | 保暖链 |
| Starving / Hunger | 饿死 | 食物链 |
| Darkness / Charlie | 黑暗击杀 | 光源链 |
| Terrorbeak / Crawling Horror / Shadow Creature | 暗影生物击杀 | 光源链 + 理智恢复不足 |
| Overheating | 过热死亡 | 夏季降温链 |
| Fire / Burning / Burnt | 火灾或燃烧 | 夏季防火链 / 雷击 / 误操作 |
| Frog | 青蛙/青蛙雨 | 春季雨湿链 + 战斗安全 |
| Lightning | 雷击 | 春季雨湿链 + 基地安全 |
| Hound / Blue Hound / Red Hound | 猎犬击杀 | 战斗安全 + 当季环境链 |
| Deerclops | 鹿角怪击杀 | 冬季基地安全 + 保暖/光源 |
| Dragonfly | 龙蝇击杀 | 夏季战斗安全 + 防火 / 降温（新手以躲避为主） |
| Pig / Spider / Tentacle / Beefalo / Killer Bee / Pengull / Mosquito | 普通生物击杀 | 探索/战斗上下文，回查食物、光源和当季环境 |

原则：

- 不要只翻译死因。
- 要说明“最后一击”和“真正断链”的区别。
- 怪物击杀通常不是单纯“不会打”，常常是食物、光源、保暖、雨湿、过热或路线问题把玩家逼到危险状态。

## 用户介入策略

读图诊断完成后，默认一次性返回两层结果：

1. 正常攻略路线。
2. 对应的物资补救代码。

其他更强、可以直接修改游戏数值的代码不在首轮直接展示，只在用户明确追问时再给。

### 策略 A：只给攻略

默认策略。

输出：

- 当前最大风险
- 根因链解释
- 下一局 3-7 步最低修正方案
- 不主动展示控制台代码

### 策略 B：给物资代码

适用：

- 用户想降低难度
- 用户不想重开太多次
- 用户希望保留探索和操作，只补齐关键资源

输出：

- 先解释这会降低生存压力
- 再给与死因对应的最小物资包

例如冬季冻死：

```lua
c_give("heatrock",1)
c_give("winterhat",1)
c_give("torch",3)
c_give("log",40)
c_give("cutgrass",40)
c_give("twigs",40)
```

### 策略 C：改属性/开挂

适用：

- 用户明确想跳过卡点
- 用户选择“开挂躺平”
- 用户在看过攻略和物资补救代码后，仍明确表示当前难度太高，需要修改游戏数值

输出：

```lua
c_sethealth(1)
c_sethunger(1)
c_setsanity(1)
```

## 选择题诊断路线（卡片式）

以下情况启用：

- 用户在开场卡片里选择“做几道选择题”。
- 用户已经提供第几天、怎么死、当时缺什么资源等信息（可直接替用户预填对应卡片，再让其确认/补选）。
- 上传截图或可选自动获取失败后，用户改用选择题。

不做自由文字长问卷，全程卡片点选。依次给出下面三张卡片（对应 `data/agent_cards.json` 的 `choice_diagnosis_cards`），用户点完自动汇总分析；每张卡都可点“说不清”跳过。

### 卡片 1：你通常卡在哪个季节？（单选）

秋季（开局/建设期）/ 冬季（冷+长夜）/ 春季（多雨潮湿）/ 夏季（过热自燃）/ 说不清（或裸本体只有冬夏）。
作用：锁定当季环境链。

### 卡片 2：哪个数值最保不住？（可多选）

血量 / 饱食度 / San 值（理智）/ 体温 / 湿度 / 说不清。
作用：指向先断的生存链——

- 饱食度 → 食物链
- San 值 → 光源链 + 理智恢复
- 体温 → 冬/春初看保暖链，夏季看降温链
- 湿度 → 春季雨湿链
- 血量 → 多为结果，结合其他项反推是被冻/被打/断粮/暗影

### 卡片 3：最大的卡点是哪一类？（可多选）

物资收集难 / 食物不够 / 战斗打不过 / 探索找路难 / 换季就崩 / 黑夜理智撑不住 / 说不清。
作用：指向被卡的能力线——

- 物资收集难 → 基地/采集链（科技、二本、工具装备）
- 食物不够 → 食物链
- 战斗打不过 → 战斗安全链（常是被环境逼到危险状态）
- 探索找路难 → 光源链 + 基地选址 + 探图节奏
- 换季就崩 → 结合季节卡的过季准备链
- 黑夜理智撑不住 → 光源链 + 理智恢复

### 汇总分析

把三张卡的选择拼成一份画像：季节锚定当季环境链，数值指向先断的生存链，卡点指向被卡的能力线；取交集得到 1 条主断链 + 1-2 条次断链，再查 `death_causes.json` / `seasonal_survival.json`，套用「固定输出模板 / 固定输出结构」输出。

冲突处理：多选相互矛盾或都选“说不清”时，以季节卡为主锚点，最多再补问**一句**（最近一次大概第几天、怎么没的），不要展开长问卷。

## 数据文件

本 Skill 调用以下规则文件：

- `data/morgue_ocr_rules.json`：OCR 字段、死因映射、取材料引导与可选自动获取策略。
- `data/death_causes.json`：停尸房 Cause of Death 到根因链、季节风险和补救策略的集中映射。
- `data/seasonal_survival.json`：四季链路和阶段判断。
- `data/agent_cards.json`：用户介入策略和兜底问答。
- `data/rules.json`：冬季/首冬三链细化规则。
- `data/facts.json`：物品、季节和版本事实库。

调用关系：`death_causes.json` 是死因判定入口，`seasonal_survival.json` 是四季上下文，`rules.json` 只在冬季/首冬相关时下钻。
