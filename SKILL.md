# Mavis Skill：饥荒停尸房截图死因分析

## 目标

当用户想知道“我在《饥荒》里为什么死了”时，默认先由 App Agent 自动检测 Steam 和《饥荒》安装，并尝试进入停尸房/讣告截图分析，而不是先要求用户上传截图或回答一串问题。

Skill 的核心输出是：

- 生存天数
- 停尸房直接死因
- 大致阶段/季节判断
- 根因链判断
- 下一局最低修正方案
- 可选援助：攻略、物资代码、属性/开挂代码

截图上传和手动问答都只是兜底：不能自动打开游戏、不是 Steam 安装、OCR 失败、或用户拒绝自动操作时才启用。用户正常点开 Skill/case 时，不先问截图，直接进入 App Agent 检测。

## 触发语义

当用户表达以下任一意图时，启用本 Skill：

- “帮我看看饥荒怎么死的”
- “分析停尸房/讣告/死亡记录”
- “打开饥荒看看我死因”
- “我又死在冬天/春天/夏天了”
- 用户上传《饥荒》停尸房截图
- 用户询问《饥荒》死亡原因、生存天数、下一局怎么改

如果用户只是泛泛问攻略，不需要分析个人死亡记录，可以走普通攻略路线；如果用户触发的是本 Skill/case，默认进入 App Agent 自动检测。

## 总体原则

1. 自动检测优先：用户触发 Skill 后，默认第一步就是让 App Agent 检测 Steam 和游戏安装。
2. App Agent 负责打开游戏和截图，Mavis 负责读图 OCR，本 Skill 负责推理。
3. 先输出可从截图读到的硬信息：生存天数、直接死因。
4. 再解释“直接死因”和“根因链”的区别。
5. 用户介入能力保留，但放在诊断之后。

## 角色分工

### Mavis 本体

负责对话编排、图片理解和 OCR。

Mavis 应把截图 OCR 成结构化字段：

```json
{
  "days_lived": 23,
  "deceased": "Wilson",
  "cause_of_death": "Darkness",
  "mode": "Survival",
  "ocr_confidence": 0.92,
  "raw_ocr_text": "..."
}
```

最低成功条件：

- `days_lived` 可读
- `cause_of_death` 可读

`deceased` 和 `mode` 可选，读不到时降置信但不阻塞。

### App Agent

负责本地应用检测、打开游戏、导航到停尸房/讣告页面并截图。

App Agent 不负责解释死因。

### 本 Skill

负责：

- 规定调用顺序
- 校验 OCR 字段
- 用天数判断阶段
- 用死因映射根因链
- 输出攻略或控制台援助

## 主流程

### 0. 用户触发

用户触发 Skill 后，默认不先判断用户有没有截图，而是直接进入 App Agent 自动检测：

- 第一步：App Agent 检测 Steam。
- 第二步：检测 Steam 库里是否安装 Don't Starve 单机版或 Don't Starve Together 联机版。
- 第三步：检测到游戏后，按单机/联机路径自动打开停尸房/讣告并截图。
- 第四步：只有未检测到 Steam/游戏、自动化失败、或产品环境不允许自动化时，才请求用户上传停尸房/讣告截图。
- 第五步：用户没有截图时，进入手动兜底。

推荐开场话术：

> 我先帮你检测 Steam 里的《饥荒》，如果能找到游戏，我会自动打开停尸房/讣告截图分析。找不到的话，你也可以上传截图，我再继续分析。

如果产品默认已经授权 App Agent 自动操作，不需要先征求“要不要自动操作”，也不需要先问用户有没有截图；直接检测 Steam，失败时再向用户说明并请求截图。

### 1. 检测 Steam

App Agent 先检测 Steam 是否安装。

优先检测：

- macOS：`/Applications/Steam.app`、`~/Applications/Steam.app`
- Windows：常见 Steam 安装路径和注册表
- Steam library manifest：`steamapps/libraryfolders.vdf`
- Steam URL scheme：`steam://`

结果分流：

- 未检测到 Steam：不要继续找游戏包名，进入非 Steam 兜底。
- 检测到 Steam：继续检测游戏安装。

未检测到 Steam 时的推荐话术：

> 我没有检测到 Steam 版《饥荒》的可自动操作入口。你可以上传停尸房截图，我照样能分析；如果不想截图，我也可以用两轮问题做兜底诊断。

### 2. 检测 Steam 游戏安装

Steam 版游戏以 app id 为准，不靠模糊包名。

目标游戏：

| 游戏 | Steam app id | 用途 |
| --- | --- | --- |
| Don't Starve | `219740` | 单机版 |
| Don't Starve Together | `322330` | 联机版 |

检测方式：

1. 从 Steam 的 `libraryfolders.vdf` 找所有库目录。
2. 查找 `appmanifest_219740.acf` 和 `appmanifest_322330.acf`。
3. 如果 manifest 存在，认为该 Steam 游戏已安装或至少在库中有可用入口。
4. 如果 manifest 不存在，不要猜非白名单包名。

结果分流：

- 只检测到 `219740`：走单机版自动化路径。
- 只检测到 `322330`：走联机版自动化路径。
- 两个都检测到：问用户要分析哪一个。
- 两个都没检测到：先请求用户上传截图；用户没有截图时才进入手动兜底。

两个都安装时的推荐话术：

> 我检测到你同时装了《饥荒》单机版和《饥荒联机版》。你想分析哪一个的死亡记录？

### 3A. 单机版自动化路径：Don't Starve

适用：Steam app id `219740`。

目标：进入单机版停尸间并截图。

推荐 UI 路径：

1. 启动 Don't Starve。
2. 到主菜单。
3. 点击「开始游戏」。
4. 点击「停尸间」/ Morgue。
5. 等待停尸房列表稳定显示。
6. 截图。
7. 把截图交给 Mavis OCR。

如果中途 UI 文案语言不同：

- “开始游戏”可对应 `Play` / `Start Game`
- “停尸间”可对应 `Morgue`

如果打不开游戏、卡启动、弹出更新或登录提示：

- 停止自动化。
- 告知用户需要手动打开停尸间并上传截图。

### 3B. 联机版自动化路径：Don't Starve Together

适用：Steam app id `322330`。

目标：进入联机版讣告并截图。

推荐 UI 路径：

1. 启动 Don't Starve Together。
2. 到主菜单。
3. 点击「资料」/ Profile。
4. 点击「讣告」/ Morgue / Obituaries。
5. 等待讣告列表稳定显示。
6. 截图。
7. 把截图交给 Mavis OCR。

如果遇到登录、离线、版本更新、隐私弹窗：

- 不强行继续点击。
- 让用户手动进入「资料 -> 讣告」后上传截图。

## 截图分析规则

Mavis OCR 后，本 Skill 按以下顺序分析：

1. 判断截图是否为停尸房/讣告页面。
2. 抽取所有可见死亡记录，不只读最新一条：`Days Lived`、`Cause of Death`、`Mode`、角色/头像。
3. 先做综合画像：统计死亡天数集中在哪些区间、重复出现的死因、重复断裂的根因链。
4. 判断“最近一次死亡”和“历史高频问题”是否一致：最近一次用于解释当下，历史模式用于给下一局优先级。
5. 输出综合结论：用户大概常死在什么时候、最常见死因是什么、背后的主要断链是什么。
6. 再拿综合结论查 `death_causes.json`、`seasonal_survival.json`，必要时下钻 `rules.json`。
7. 生成下一局最低修正方案：优先解决高频/早期/最阻断进度的死因，而不是只修最新一条。
8. 根据用户介入策略输出正常攻略路线和对应物资补救代码。

多记录分析原则：

- 如果截图有多条记录，必须先列出可读记录摘要，再做综合判断。
- 不要默认只分析最上方/最新记录；最上方记录只是“最近一次死亡”。
- 如果某类死因重复出现，优先级高于单次偶发死因。
- 如果死亡天数集中在同一窗口，优先判断该阶段的系统性短板。
- 如果最新一次和历史高频不一致，输出时分开说：“最近一次是 X，但历史上更常见的问题是 Y。”
- 如果 OCR 只读清部分记录，也要说明“基于可读记录分析”。

规则表调用顺序必须固定：

1. 先形成多记录综合结论：死亡天数分布、死因分布、根因链分布。
2. 再查 `data/death_causes.json`：用综合后的高频/最近 `Cause of Death` 确定直接死因、根因链、季节风险和基础补救方向。
3. 再查 `data/seasonal_survival.json`：根据根因链和天数/季节上下文，补充四季层面的环境链判断。
4. 只有当死因或上下文强相关冬季/首冬时，才下钻 `data/rules.json`：例如 `Freezing`、`Deerclops`、冬季长夜黑暗、冬季断粮、蓝猎犬、暖石/冬帽/火源/燃料问题。
5. 不要把 `data/rules.json` 当成总入口；它只是冬季放大镜。

截图不是停尸房时：

> 这张图不像停尸房/讣告页面，我读不到稳定的生存天数和死因。你可以进入停尸房再截一张；或者我用手动问题帮你判断。

OCR 低置信时：

- 最多补问 1 个问题。
- 不进入长问卷。

推荐补问：

> 我能看出生存天数大概是第 X 天，但死因那一栏不太清楚。你能确认一下 Cause of Death 写的是什么吗？

## 天数到阶段的粗略判断

仅当默认采用 RoG/Giant Edition 秋季起始时使用：

| 生存天数 | 推断阶段 |
| --- | --- |
| 1-5 | 开局基础生存失败 |
| 6-14 | 秋季建设窗口 |
| 15-20 | 首冬准备窗口 |
| 21-36 | 冬季/首冬窗口 |
| 37-56 | 春季雨湿窗口 |
| 57-72 | 夏季过热/防火窗口 |
| 73+ | 后续循环或后期生存 |

注意：

- RoG 可能春季起始。
- DST 季节长度和单机有差异。
- 用户可能改世界设置。
- 裸 Don't Starve 本体不是四季制。

因此：天数只做辅助，最终以死因和截图上下文为主。

## 死因到根因链

| Cause of Death | 直接含义 | 根因链 |
| --- | --- | --- |
| Freezing / Frozen | 冻死 | 保暖链 |
| Starving / Hunger | 饿死 | 食物链 |
| Darkness / Charlie | 黑暗击杀 | 光源链 |
| Terrorbeak / Crawling Horror / Shadow Creature | 暗影生物击杀 | 光源链 + 理智恢复不足 |
| Overheating | 过热死亡 | 夏季降温链 |
| Fire / Burning / Burnt | 火灾或燃烧 | 夏季防火链 / 雷击 / 误操作 |
| Frog | 青蛙/青蛙雨 | 春季雨湿链 + 战斗安全 |
| Lightning | 雷击 | 春季雨湿链 + 基地安全 |
| Hound / Blue Hound / Red Hound | 猎犬击杀 | 战斗安全 + 当季环境链 |
| Deerclops | 鹿角怪击杀 | 冬季基地安全 + 保暖/光源 |
| Pig / Spider / Tentacle / Beefalo / Killer Bee / Pengull | 普通生物击杀 | 探索/战斗上下文，回查食物、光源和当季环境 |

原则：

- 不要只翻译死因。
- 要说明“最后一击”和“真正断链”的区别。
- 怪物击杀通常不是单纯“不会打”，常常是食物、光源、保暖、雨湿、过热或路线问题把玩家逼到危险状态。

## 用户介入策略

读图诊断完成后，默认一次性返回两层结果：

1. 正常攻略路线。
2. 对应的物资补救代码。

其他更强、可以直接修改游戏数值的代码不在首轮直接展示，只在用户明确追问时再给。

### 策略 A：只给攻略

默认策略。

输出：

- 当前最大风险
- 根因链解释
- 下一局 3-7 步最低修正方案
- 不主动展示控制台代码

### 策略 B：给物资代码

适用：

- 用户想降低难度
- 用户不想重开太多次
- 用户希望保留探索和操作，只补齐关键资源

输出：

- 先解释这会降低生存压力
- 再给与死因对应的最小物资包

例如冬季冻死：

```lua
c_give("heatrock",1)
c_give("winterhat",1)
c_give("torch",3)
c_give("log",40)
c_give("cutgrass",40)
c_give("twigs",40)
```

### 策略 C：改属性/开挂

适用：

- 用户明确想跳过卡点
- 用户选择“开挂躺平”
- 用户在看过攻略和物资补救代码后，仍明确表示当前难度太高，需要修改游戏数值

输出：

```lua
c_sethealth(1)
c_sethunger(1)
c_setsanity(1)
c_godmode()
```

提醒：

- 首轮输出不要直接展示这些代码。
- 首轮固定提示文案：

> 我暂时不直接展示其他更强、可以直接修改游戏数值的代码。  
> 如果你觉得当前难度还是太高，确实需要的话，也可以再问我要；不过这类代码会明显影响游戏体验，建议想清楚后再问我哦～我会给你并提醒每条代码的影响。

- 属性代码不解决环境链。
- 冻结、过热、黑暗、火灾仍可能破坏体验。
- 建议至少保留对应物资方案。

## 手动兜底路线

只有以下情况启用：

- 未检测到 Steam。
- Steam 中没有检测到 `219740` 或 `322330`。
- 用户使用非 Steam 版本，App Agent 不在白名单内。
- 用户不想自动截图。
- 截图/OCR 失败。

兜底最多两轮问题。

第一轮：

> 你大概活到第几天？最后显示的死因是什么，或者你记得是冻死、饿死、黑暗、怪物、过热、火灾哪一种？

第二轮按死因补问：

- 冻死：有没有暖石/火源/保暖衣？
- 饿死：有没有锅/稳定食物来源？
- 黑暗/暗影：有没有火把/夜间行动计划/理智恢复？
- 过热：有没有冷火/冷暖石/防暑装备？
- 火灾：基地有没有防火方案？
- 青蛙/雷击：是否春季雨天，有没有避雷/避战？

不要问完整资源清单。

综合看，你最需要先补的是：某某链。

为什么：
解释高频死因如何从断链转成死亡；如果最近一次和历史高频不同，要分开说明。

下一局最低修正：
1. ...
2. ...
3. ...

```

## 失败与降级

如果 Steam 不存在：

> 我没有检测到 Steam 入口，所以不能稳定自动打开游戏。你可以上传停尸房截图；如果不想截图，我问两个问题也能给你一个低置信诊断。

如果两个游戏都没检测到：

> 我没有在 Steam 库里检测到《饥荒》单机版或联机版。可能是非 Steam 版本、未安装或不在白名单内。你可以直接上传停尸房截图。

如果两个游戏都检测到：

> 我检测到你同时装了单机版和联机版。你想分析哪一个？

如果截图失败：

> 我没能稳定进入停尸房/讣告页面。你可以手动进入后截图，我继续分析。

## 数据文件

本 Skill 调用以下规则文件：

- `data/morgue_ocr_rules.json`：OCR 字段、死因映射、自动检测优先策略。
- `data/death_causes.json`：停尸房 Cause of Death 到根因链、季节风险和补救策略的集中映射。
- `data/seasonal_survival.json`：四季链路和阶段判断。
- `data/agent_cards.json`：用户介入策略和兜底问答。
- `data/rules.json`：冬季/首冬三链细化规则。
- `data/facts.json`：物品、季节和版本事实库。

调用关系：`death_causes.json` 是死因判定入口，`seasonal_survival.json` 是四季上下文，`rules.json` 只在冬季/首冬相关时下钻。
